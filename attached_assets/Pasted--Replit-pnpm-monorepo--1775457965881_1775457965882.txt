【API Server 修改】
目标文件优先检查并修改：

一、统一鉴权与统一错误处理
1. 在 /v1 路由范围内统一使用 requireProxyBearer 中间件。
2. Bearer Token 必须严格校验 Authorization 头，格式为：
   Authorization: Bearer ${PROXY_API_KEY}
3. 缺少 token 或 token 错误时，所有 /v1 接口都返回：
   HTTP 401
   Content-Type: application/json
   JSON 结构：
   {
     "error": {
       "message": "Unauthorized: invalid or missing Bearer token",
       "type": "authentication_error"
     }
   }
4. 所有参数校验失败、模型不支持、上游错误、内部异常，都必须返回 JSON，不能返回 HTML。
5. 增加统一 JSON 错误处理中间件，覆盖 Express 默认错误页。
6. /v1 下不存在的路径要返回 JSON 404，不要落回门户 HTML。

二、app.ts 挂载方式
在 artifacts/api-server/src/app.ts 中确保：

五、POST /v1/chat/completions
在 proxyRouter 中实现相对路径 POST /chat/completions。
这是 OpenAI 兼容接口，要求支持 OpenAI 模型和 Claude 模型。

基础规则：
1. 先鉴权，再校验 body，再调用 provider。
2. 请求 Content-Type 为 application/json。
3. 缺少 model 时返回 JSON：
   {
     "error": {
       "message": "model is required",
       "type": "invalid_request_error"
     }
   }
4. model 不在支持列表时返回 JSON 400，明确说明 unsupported model。
5. 按模型分发：
   - model 以 gpt- 或 o 开头 -> OpenAI provider
   - model 以 claude- 开头 -> Anthropic provider

OpenAI provider 路径：
1. 使用 OpenAI SDK 调用真实 chat/completions 能力；不要猜 API，必须按当前安装版本的真实 SDK 写。
2. 非流式时返回 OpenAI 原始 JSON 或严格兼容 JSON。
3. 流式时返回标准 SSE，直接转发或兼容转发 OpenAI chunk。
4. 出错时捕获并转成 JSON 错误，不得返回 HTML。

Anthropic provider 路径，需要把 OpenAI Chat Completions 请求格式转换成 Anthropic Messages 格式：
1. 消息转换：
   - 普通 user / assistant / system 文本消息正确映射
   - role: "tool" 转换为 Anthropic tool_result 内容块
   - assistant 中的 tool_calls 转换为 tool_use 内容块
2. tools 转换：
   - OpenAI tools[].function.parameters -> Anthropic input_schema
3. tool_choice 转换：
   - required -> any
   - auto -> auto
   - 指定函数名时映射为 Anthropic 对应选择方式；如果 SDK 需要明确结构，按真实 SDK 写
4. 响应转换：
   - Anthropic 文本响应转换为 OpenAI chat completion JSON
   - Anthropic tool_use 块转换为 OpenAI tool_calls
   - stop_reason: "tool_use" -> finish_reason: "tool_calls"
5. 非流式时：
   - 如果当前 Anthropic SDK/平台限制要求内部使用 stream().finalMessage()，可以这样做
   - 但最终返回给客户端必须是合法 OpenAI chat completion JSON
6. 流式时：
   - 返回标准 SSE
   - Header 至少包含：
     - Content-Type: text/event-stream
     - Cache-Control: no-cache, no-transform
     - Connection: keep-alive
     - X-Accel-Buffering: no
   - 调用 res.flushHeaders()
   - 把 Anthropic 事件流转换为 OpenAI chat.completion.chunk 格式
   - 若出现工具调用增量，正确拼接到 function.arguments
   - 每 5 秒发送一次 `: keepalive\n\n`
   - req.on("close") 时及时清理 keepalive 和上游流
   - 必须用 try/finally 防止连接断开时产生 500

六、POST /v1/messages
在 proxyRouter 中实现相对路径 POST /messages。
这是 Anthropic Messages 原生接口，要求支持 Claude 模型和 OpenAI 模型。

基础规则：
1. 先鉴权，再校验 body，再调用 provider
2. 接受 Anthropic Messages API 原生请求结构：
   - system
   - messages
   - tools
   - tool_choice
   - max_tokens
   - stream
   - 以及透传的 cache_control
3. model 不支持时返回 JSON 400

Claude provider 路径：
1. 尽可能直接透传到 Anthropic SDK
2. 非流式时返回合法 Anthropic Message JSON
3. 流式时透传或兼容转发 Anthropic SSE 事件，至少保证：
   - message_start
   - content_block_start
   - content_block_delta
   - content_block_stop
   - message_delta
   - message_stop
   - ping
4. 出错时返回 JSON，不要返回 HTML

OpenAI provider 路径，需要把 Anthropic Messages 请求格式转换成 OpenAI 请求格式：
1. messages 转换：
   - text -> OpenAI message content
   - tool_result -> role: "tool"
   - tool_use -> assistant tool_calls
2. tools 转换：
   - input_schema -> function.parameters
3. tool_choice 转换到 OpenAI 兼容结构
4. 响应转换：
   - OpenAI 响应 -> Anthropic Message JSON
   - content[]、stop_reason、usage.input_tokens/output_tokens 结构正确
5. 流式时：
   - 将 OpenAI 流式输出合成为 Anthropic SSE 事件序列
   - 工具调用时生成对应 tool_use 事件块
   - 保持事件顺序正确

七、cache 机制
1. Anthropic 原生支持 cache_control，代理必须透传 system 和 messages 中的 cache_control 字段，不要丢失。
2. OpenAI gpt-5 系列的 prompt prefix cache 是提供商原生行为，无需额外注入参数。
3. 前端门户中可以说明这件事，但不要伪造代理层实现不存在的缓存逻辑。

2. 记录 endpoint、provider、model、stream、status code、耗时
3. 出错时记录上游 provider 的状态码和错误消息
4. 不要打印密钥
